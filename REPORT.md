# Отчёт по проекту Issue Board

## Суть приложения

Issue Board — это веб-приложение, которое представляет GitHub Issues в виде канбан-доски с тремя колонками: **To Do**, **In Progress** и **Done**. Пользователь может перетаскивать карточки между колонками, создавать новые issue, а все изменения синхронизируются с GitHub через REST API.

Ключевая особенность — **offline-first подход**: приложение работает без интернета, сохраняет все действия в локальный лог операций и автоматически синхронизирует их при восстановлении соединения. Также реализован механизм обнаружения и разрешения конфликтов, когда один и тот же issue был изменён и локально, и на GitHub.

Приложение оформлено как **PWA** (Progressive Web App), то есть его можно установить на рабочий стол и пользоваться как нативным.

## Архитектура: Feature-Sliced Design (FSD)

Проект построен по методологии **Feature-Sliced Design** — архитектурному стандарту для фронтенд-приложений, который задаёт чёткие правила разбиения кода на слои и слайсы.

### Структура слоёв

```
src/
├── app/              # Инициализация приложения (провайдеры)
│   └── provider/     # QueryProvider (React Query)
├── pages/            # Страницы (точки входа маршрутов)
│   ├── BoardPage     # Главная страница с доской
│   └── SettingsPage  # Страница настроек
├── features/         # Пользовательские сценарии
│   ├── board/        # Канбан-доска, колонки, DnD, создание issue
│   ├── filters/      # Поиск и фильтрация по state/labels
│   ├── settings/     # Форма настроек (репозиторий, токен)
│   └── sync/         # Синхронизация с GitHub, операционный лог
├── entities/         # Бизнес-сущности
│   ├── issue/        # Issue: схема, стор, API, маппер, карточка
│   └── label/        # Label: схема, UI
└── shared/           # Переиспользуемые утилиты
    ├── lib/          # useOnlineStatus, вспомогательные функции
    └── ui/           # Общие UI-компоненты (Card, Select)
```

### Принципы

- **Слои** идут сверху вниз по уровню абстракции: `app` → `pages` → `features` → `entities` → `shared`. Каждый слой может импортировать только из нижестоящих.
- **Слайсы** внутри слоя — это изолированные модули по бизнес-домену (например, `board`, `sync`, `issue`).
- **Сегменты** внутри слайса делят код по назначению: `model/` (стор, схемы, типы), `ui/` (компоненты), `api/` (запросы к GitHub), `lib/` (хелперы).

### Почему FSD

Без архитектурных правил код быстро превращается в кашу: компоненты импортируют друг друга хаотично, бизнес-логика размазана по файлам. FSD решает эту проблему — сразу понятно, где искать стор issue (`entities/issue/model/`), где логику синхронизации (`features/sync/model/`), а где UI доски (`features/board/ui/`).

**Аналоги:** Atomic Design, классическая структура по типам файлов (`components/`, `hooks/`, `utils/`). Atomic Design хорошо подходит для UI-библиотек, но не даёт правил для бизнес-логики. Группировка по типам файлов (все компоненты в одной папке) не масштабируется — когда компонентов становится 50+, ориентироваться в плоском списке невозможно. FSD масштабируется лучше всего.

## Используемые технологии

### React 19

React — основная UI-библиотека проекта. Выбрана, потому что это самый популярный инструмент для создания интерактивных интерфейсов с компонентным подходом. Используется компонентная архитектура с функциональными компонентами и хуками (`useState`, `useEffect`, `useMemo`, `useRef`).

**Аналоги:** Vue.js, Svelte, Angular. Vue тоже хороший вариант, но у React больше экосистема библиотек и больше ресурсов для обучения. Angular показался слишком громоздким для проекта такого размера. Svelte интересный, но менее распространён в продакшене.

### TypeScript 5.9

TypeScript добавляет статическую типизацию поверх JavaScript. В проекте используется для типизации пропсов компонентов, состояния сторов, схем данных и ответов API. Это сильно помогает при рефакторинге — компилятор сразу показывает, где сломалась логика.

**Аналоги:** чистый JavaScript, Flow. Flow практически мёртв, а на чистом JS в проекте такого размера легко наделать ошибок, которые проявятся только в рантайме. TypeScript — стандарт индустрии.

### Vite 7.3

Vite — сборщик и dev-сервер. Выбран за мгновенный запуск dev-сервера (благодаря нативным ES-модулям) и быструю сборку через Rolldown. В конфиге подключены плагины для React (HMR), Tailwind CSS и PWA.

**Аналоги:** Webpack, Parcel, Turbopack. Webpack — проверенный вариант, но конфигурация сложнее и dev-сервер медленнее. Parcel проще, но менее гибкий. Vite оказался идеальным балансом между скоростью и простотой настройки.

### Tailwind CSS 4.2

Tailwind — утилитарный CSS-фреймворк. Стили пишутся прямо в JSX через классы (`className="flex items-center gap-2 ..."`). В проекте используется Tailwind v4 с CSS-переменными для темизации (oklch-цвета).

**Аналоги:** обычный CSS / CSS Modules, styled-components, Sass. CSS Modules хороши для изоляции, но нужно постоянно переключаться между файлами. styled-components удобен, но увеличивает бандл и даёт оверхед в рантайме. Tailwind позволяет писать стили быстро, не покидая компонент.

### Zustand 5.0

Zustand — минималистичный стейт-менеджер. В проекте три стора: `issues.store` (список issue), `filters.store` (фильтры), `settings.store` (настройки репозитория и токена). Для `issues` и `settings` используется middleware `persist`, чтобы данные сохранялись в `localStorage`.

**Аналоги:** Redux Toolkit, MobX, Jotai, Context API. Redux мощный, но очень много бойлерплейта (слайсы, экшены, редьюсеры). MobX использует декораторы и прокси, что усложняет понимание. Context API подходит для простых случаев, но при частых обновлениях вызывает лишние ререндеры. Zustand — это буквально `create((set) => ({...}))` и всё работает.

### React Router 7.13

React Router обеспечивает клиентскую маршрутизацию. В приложении два маршрута: `/` — доска с issue, `/settings` — страница настроек. Используются компоненты `BrowserRouter`, `Routes`, `Route` и `Link`.

**Аналоги:** TanStack Router, Next.js App Router. TanStack Router типобезопасный, но ещё молодой. Next.js — целый фреймворк, для SPA-проекта это перебор. React Router — классика, хорошо документирован.

### TanStack React Query 5.90

React Query управляет асинхронными операциями. В проекте используется `useMutation` для синхронизации: push локальных изменений на GitHub и pull удалённых issue. Обеспечивает удобное отслеживание состояний `isPending`, `isSuccess`, `isError`.

**Аналоги:** SWR, самописные хуки с `useEffect` + `fetch`. SWR чуть проще, но React Query мощнее в плане мутаций, инвалидации кэша и девтулзов. Самописные хуки быстро превращаются в спагетти при обработке ошибок и ретраев.

### @dnd-kit/react 0.3

dnd-kit — библиотека для drag & drop. Используются хуки `useDraggable` (на карточках issue) и `useDroppable` (на колонках). При дропе карточки в другую колонку срабатывает `changeIssueStatus`.

**Аналоги:** react-beautiful-dnd, react-dnd. react-beautiful-dnd больше не поддерживается и несовместим с React 19. react-dnd мощный, но API сложнее. dnd-kit — современная, лёгкая и хорошо работает с React 19.

### React Hook Form 7.71 + Zod 4.3

React Hook Form управляет формой настроек (репозиторий и токен). Zod используется для валидации через `zodResolver` — схема описывает правила (обязательные поля, формат `owner/repo`), а форма автоматически показывает ошибки.

**Аналоги:** Formik + Yup, нативная валидация. Formik — тяжелее и рендерит форму целиком при каждом нажатии клавиши. React Hook Form использует `ref`-подход и минимизирует ререндеры. Yup — аналог Zod, но Zod лучше работает с TypeScript (выводит типы из схемы).

### vite-plugin-pwa 1.2 + Workbox

Плагин превращает приложение в PWA. Генерирует `manifest.webmanifest`, Service Worker (через Workbox) и обеспечивает кэширование. Для GitHub API настроена стратегия `NetworkFirst` — сначала пробует сеть, при отсутствии отдаёт кэш.

**Аналоги:** ручная настройка Service Worker, @vite-pwa/assets-generator. Ручная настройка SW — долго и ошибкоёмко. vite-plugin-pwa делает 90% работы автоматически: precache статики, рантайм кэширование, авто-обновление.

### Lucide React 0.577

Lucide — набор SVG-иконок в виде React-компонентов. Используются иконки для кнопок (Settings, RefreshCw, Plus, WifiOff и др.). Каждая иконка — отдельный импорт, что обеспечивает tree-shaking.

**Аналоги:** React Icons, Heroicons, FontAwesome. React Icons включает сразу все наборы и может раздуть бандл. FontAwesome требует подключения шрифта. Lucide — лёгкие, красивые, со стилем Feather Icons.

## Заключение

Работа над Issue Board заняла больше времени, чем я ожидала, особенно на отладку синхронизации. Самым сложным оказался offline-first подход: нужно было продумать операционный лог, ретраи с экспоненциальной задержкой, обработку удалённых на GitHub issue (404/410), обновление временных ID после создания и разрешение конфликтов. Каждый раз, когда казалось, что всё работает, появлялся новый edge-case.

Из технологий больше всего понравился **Zustand** — после Redux это как глоток свежего воздуха. Буквально 20 строк кода — и у тебя полноценный стор с персистентностью. **Tailwind** тоже оказался удобнее, чем я думала: сначала классы кажутся нечитаемыми, но через пару часов привыкаешь и перестаёшь переключаться между CSS-файлами.

**React Hook Form + Zod** — отличная связка. Раньше валидацию форм я писала руками через `useState` и кучу `if`-ов. Здесь описываешь схему один раз — получаешь и типы, и валидацию, и сообщения об ошибках.

**dnd-kit** работает хорошо, но документация могла бы быть лучше, особенно для новой React-версии библиотеки. Пришлось разбираться по примерам.

В целом проект дал понимание того, как устроены реальные offline-first приложения с синхронизацией. Это сильно отличается от обычных CRUD-приложений, где просто отправляешь запрос и ждёшь ответ. Здесь нужно думать о конфликтах, очередях, идемпотентности операций и пользовательском опыте при отсутствии сети.
